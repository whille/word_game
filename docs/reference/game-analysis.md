## 二、实现分析

### 2.1 核心架构（推测）

```
┌─────────────────────────────────────────┐
│              关卡数据层 (JSON)             │
│  nodes: Node[]  rules: Rule[]  items: Item[] │
├─────────────────────────────────────────┤
│              状态管理层                     │
│  健康值 / 理智值 / 道具背包 / 节点访问历史    │
├─────────────────────────────────────────┤
│              规则引擎                      │
│  条件表达式求值 → 节点可见性/可达性判定       │
├─────────────────────────────────────────┤
│              UI 渲染层                     │
│  节点树渲染 / 连线绘制 / 道具拖拽 / 特效     │
└─────────────────────────────────────────┘
```

### 2.2 节点系统

节点是游戏的基本交互单位，类型体系：

| 节点类型 | 视觉表现 | 用途 |
|---------|---------|------|
| 基础选项 | 米黄色纸条 | 场景/动作描述 |
| 拓展选项 | 纸条 + 放大镜/扳手/开关图标 | 需要道具交互 |
| 线索选项 | 白色笔记本残页 + 放大镜/问号 | 关键信息提示 |
| 怪物节点 | 暗红色纸条 + 血滴图标 | 可击杀目标 |
| 玩家位置 | 白色小人图标 | 标记当前位置 |

节点之间由**粉笔线**连接，形成树形/网状拓扑。已探索节点变黯淡。

```json

```

### 2.4 交互模型

| 操作 | 实现 |
|------|------|
| 点击节点 | 展开子节点，推进叙事 |
| 拖拽道具 | 物品栏拖到目标节点，高亮相容节点（蓝色线框） |
| 右键 | 关闭笔记本/线索弹窗 |
| 道具组合 | 子弹 + 手枪 → 可射击手枪 |
| 特殊交互 | 枪械：瞄准镜 → 鼠标移动 → 射击 |

### 2.5 状态系统

- **健康值 (HP)** + **理智值 (San)**：任一归零 → 坏结局
- **道具背包**：收集/消耗/组合
- **笔记本**：自动记录已发现的规则和线索
- **节点访问历史**：已探索节点变黯淡

## 三、优点

1. **题材稀缺**：规则怪谈 × 文字交互，独一无二的品类占位
2. **直觉化操作**：无需引导，看一眼就知道怎么玩
3. **编辑器 + 工坊**：从"游戏"变成"规则怪谈创作工具"，生命周期远超普通叙事游戏
4. **音效氛围**：纯文字但靠音效和 jump scare 制造恐惧
5. **文字特效**：字体变形、排列扭曲，用排版营造"克系"恐怖
6. **留白恰当**：没有动画演出反而给了文学想象空间
7. **性价比极高**：12 元定价
8. **节点视觉语言清晰**：不同类型用不同颜色/图标区分，粉笔线连线风格统一

## 四、难点与缺点

### 4.1 玩家反馈的硬伤

| 问题 | 严重度 | 详情 |
|------|--------|------|
| **无可回溯存档** | 🔴 致命 | 多结局必须从头重玩，重复操作消磨耐心（后续更新已加） |
| **重复操作多** | 🔴 致命 | 刷全结局时大量重复点击，无跳过机制 |
| **UI 杂乱** | 🟡 中等 | 节点展开多后桌面混乱，易误触 |
| **交互精度差** | 🟡 中等 | 道具拖拽需精准卡位 |
| **内容量少** | 🟡 中等 | 仅 3 个正式关卡 |
| **剧情晦涩** | 🟡 中等 | 真结局线索逻辑不清晰，"谜语人" |
| **偏离怪谈内核** | 🟡 中等 | 部分关卡混入玄学/搞笑元素，破坏恐怖氛围 |
| **Bug** | 🟡 中等 | 道具消失等 |

### 4.2 技术难点

| 难点 | 说明 |
|------|------|
| **规则矛盾的形式化** | 如何用数据结构表达"规则 A 和规则 B 矛盾，且玩家可通过规则 C 推理出生路"？纯条件表达式不够，需要更高阶的推理判定 |
| **节点布局算法** | 树形节点在屏幕上自动排列，避免重叠和交叉连线——本质是图布局问题（Force-directed / Sugiyama） |
| **关卡编辑器的 UX** | 让非程序员能创建有逻辑的规则怪谈关卡——这是工具设计里最难的部分 |
| **内容创作的不可自动化** | 技术能搭框架，但规则和故事的质量完全依赖人工创作 |

### 4.3 设计难点

| 难点 | 说明 |
|------|------|
| **可推理 vs 太简单** | 矛盾太明显 → 没挑战；太隐晦 → 变"谜语人"。这个平衡极难 |
| **留白 vs 补全** | 规则怪谈的魅力在于不说透，但游戏需要给玩家足够的反馈来确认"我猜对了" |
| **恐怖 vs 可玩** | 太恐怖劝退轻度玩家，太轻松失去怪谈特质 |
| **多结局的代价** | 每个新分支 = 内容量翻倍。32 个结局但大部分是"死法"，真结局只有几个 |

## 五、高 ROI 改进方向

### 5.1 平台切换：Web 优先 (ROI ★★★★★)

| 维度 | Steam 版 (Unity) | Web 版 |
|------|-----------------|--------|
| 安装摩擦 | 下载 → 安装 → 启动 | 点击链接即玩 |
| 文字渲染 | Unity TextMeshPro | CSS + Web Fonts（更灵活） |
| 文字特效 | Shader 实现 | CSS animation/filter（开发成本更低） |
| 分享 | 工坊链接 | URL 直链 |
| 移动端 | 不支持 | 响应式即可 |
| 开发成本 | 高（Unity 学习曲线） | 低（HTML/CSS/JS 即可） |

**关键判断**：Steam 版没有移动端。文字恐怖游戏在手机上（被窝里看）体验极佳——这是蓝海。

### 5.2 分支快照系统 (ROI ★★★★★)

```
玩家在决策点自动创建快照 → 可视化"分支树" → 一键跳转到任意分支点
```

直接解决"刷多结局要重头玩"的 #1 痛点。

### 5.3 路径迷你地图 (ROI ★★★★)

首次通关后显示已探索节点图，点击跳转到任意已访问节点。消除重复点击。

### 5.4 规则矛盾可视化 (ROI ★★★★)

当玩家发现两条矛盾规则时，UI 高亮提示冲突。但**不解说矛盾如何化解**——让玩家自己推理。既降低了"谜语人"挫败感，又保留了推理乐趣。

### 5.5 自适应节点布局 (ROI ★★★)

自动折叠无关分支，保持当前活跃路径在视觉中心。解决"UI 杂乱"痛点。

### 5.6 创作工具增强 (ROI ★★★★)

| 功能 | 价值 |
|------|------|
| 规则一致性检查 | 自动标记"没有任何节点引用这条规则" |
| 分支覆盖可视化 | 显示哪些结局可达、哪些节点孤立 |
| Playtest 模式 | 快速遍历所有路径并生成覆盖率报告 |

## 六、工程设计建议

### 6.1 技术栈推荐

```
前端：React + TypeScript + CSS Modules
状态管理： Zustand (轻量)
布局引擎： dagre / elkjs (自动图布局)
存储： localStorage (存档) + JSON (关卡文件)
测试： Vitest + Playwright
```

选择 Web 而非 Unity 的理由：
- 零安装摩擦
- CSS 文字特效开发效率远超 Shader
- URL 分享 = 天然的"创意工坊"
- 响应式 = 同时覆盖 PC 和移动端
- 团队不需要学 Unity

### 6.2 核心模块划分

```
src/
├── engine/           # 规则引擎核心
│   ├── state.ts      # 游戏状态机 (HP/San/背包/节点历史)
│   ├── rules.ts      # 规则求值器 (条件表达式 → 节点可达性)
│   └── nodes.ts      # 节点树操作 (展开/折叠/跳转)
├── data/             # 关卡数据
│   ├── schema.ts     # 关卡 JSON Schema 定义
│   └── levels/       # 关卡 JSON 文件
│       └── tutorial.json
├── ui/               # 渲染层
│   ├── NodeTree.tsx  # 节点树渲染 + 连线
│   ├── NodeCard.tsx  # 单个节点卡片
│   ├── Inventory.tsx # 道具背包
│   ├── StatusBar.tsx # HP/San 状态条
│   └── BranchMap.tsx # 分支地图
├── editor/           # 关卡编辑器 (第二阶段)
└── App.tsx
```

### 6.3 开发阶段

| 阶段 | 目标 | 交付物 |
|------|------|--------|
| **Phase 1: MVP** | 核心交互跑通 | 节点树渲染 + 点击展开 + 状态管理 + 1 个最小关卡 |
| **Phase 2: 完整关卡** | 叙事验证 | 1 个完整关卡（7-10 条规则，多结局，15-30 min 流程） |
| **Phase 3: 分支快照** | 体验优化 | 分支快照系统 + 迷你地图 |
| **Phase 4: 编辑器** | 创作工具 | 可视化关卡编辑器 + JSON 导出/导入 |
| **Phase 5: 上线** | 部署 | Vercel/CloudFlare Pages 部署 + 响应式适配 |

### 6.4 关键设计决策（待讨论）

1. **节点布局**：自动布局 (dagre) vs 手动布局（作者指定坐标）？
   - 自动布局：省心但可能不够美观
   - 手动布局：更精细但增加创作成本
   - **建议**：默认自动 + 可选手动覆盖坐标

2. **规则引擎的表达力**：简单的条件表达式 vs DSL？
   - 条件表达式：`has_item('key') AND sanity >= 50`
   - 足够覆盖大部分场景，不需要 DSL

3. **存档粒度**：每步自动存 vs 仅决策点存？
   - **建议**：仅决策点自动快照，手动存档任意时刻
